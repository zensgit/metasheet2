# 多维表治理权限模型设计

- 日期：2026-04-18
- Worktree：`.worktrees/dingtalk-member-group-role-sync-20260418`
- 分支：`codex/dingtalk-member-group-role-sync-20260418`

## 问题

后期是否可以在多维表里把某个：

- 表
- 视图
- 列
- 行
- 单元格

分别对某些用户或权限组做：

- 隐藏
- 只读
- 可编辑
- 可管理

同时，这套模型如何与现有的：

- 钉钉部门同步
- 平台成员组
- 插件管理员
- namespace admission

联动，而不是另起一套孤立 ACL。

## 当前基础能力

从当前代码看，已经具备 4 层基础：

### 1. 表级（sheet）

已有基于 `user` / `role` 的表级授权入口。

可表达：

- `read`
- `write`
- `admin`
- `write-own`（用户限定）

### 2. 视图级（view）

已有视图级权限推导与管理面。

可表达：

- `canAccess`
- `canConfigure`
- `canDelete`

### 3. 列级（field）

已有字段级权限推导。

可表达：

- `visible`
- `readOnly`

即当前已经支持“隐藏列”和“只读列”这类能力基础。

### 4. 行级（record）

已有记录级权限。

可表达：

- `read`
- `write`
- `admin`

即当前已经具备“某条记录只允许某些主体读 / 改 / 管”的基础。

## 设计结论

### 先做强：表级 + 视图级 + 列级 + 行级

这是推荐主模型。

原因：

- 规则足够表达大多数业务权限；
- UI 可解释；
- 与现有 permission derivation / manager surfaces 一致；
- 与钉钉成员组和插件治理最容易接轨。

### 后做弱：单元格级

单元格级不要一开始就做成全量自由 ACL。

推荐把单元格级能力定义成：

- 少量高敏感例外；
- 规则驱动；
- 依附于列级 / 行级之上的补丁层。

不推荐做“任何单元格都可单独授权”，因为它会带来：

- 规则爆炸；
- 性能复杂；
- 协同编辑冲突复杂；
- UI 难解释；
- 审计回溯困难。

## 推荐的五层模型

### 第 1 层：表级

问题：用户能不能进入这张表？

建议权限：

- `no_access`
- `read`
- `write`
- `admin`
- `write_own`

### 第 2 层：视图级

问题：用户能不能看某个视图、配置某个视图？

建议权限：

- `access`
- `configure`
- `delete`

用途：

- 同一张表中，把不同业务视图开放给不同角色；
- 不改变底层表权限，只控制视图入口与配置权。

### 第 3 层：列级

问题：用户能不能看某列、改某列？

建议权限：

- `visible`
- `hidden`
- `readonly`

这是最优先值得做强的一层。

适合表达：

- 薪资列只给 HR 看；
- 审批结果列对普通成员只读；
- 内部备注列对外部协作成员隐藏。

### 第 4 层：行级

问题：用户能不能看某条记录、改某条记录？

建议权限：

- `read`
- `write`
- `admin`

适合表达：

- 只能看自己负责的记录；
- 某成员组只能看本部门记录；
- 某插件管理员只能改自己范围内的记录。

### 第 5 层：单元格级（例外层）

问题：某一列整体可见，但个别单元格是否要特殊保护？

建议只支持两类场景：

1. 高敏感例外
   - 例如“薪资列可见，但某些特殊记录单元格仅 HR manager 可见”
2. 规则驱动例外
   - 例如“状态=已封存 时，某单元格只读”

不建议支持：

- 人工逐格自由授权；
- 任意用户随手维护单元格 ACL。

## 权限主体建议

### 当前可用主体

- `user`
- `role`

### 下一步推荐扩展主体

- `platform_member_group`

原因：

- 钉钉部门已经在往 `platform_member_groups` 投影；
- 成员组是组织治理和插件治理的共同交汇点；
- 比直接把 DingTalk 部门硬接进多维表 ACL 更稳。

### 为什么不建议直接把钉钉部门当多维表权限主体

因为钉钉部门是上游目录镜像，不应该直接成为本地运行时 ACL 的唯一锚点。

更稳的路径是：

1. 钉钉部门同步
2. 投影成平台成员组
3. 多维表权限引用成员组

这样就不会把目录同步模型和业务权限模型硬绑死。

## 与插件管理员的联动

推荐关系：

- 插件管理员负责某个 namespace
- 成员组决定其可治理范围
- 多维表权限再引用相同成员组

这样可以做到：

- 插件管理员只管理自己范围内的人；
- 多维表行级 / 列级权限也按同样的人群边界生效；
- 组织治理与插件治理共享一套主体模型。

## 典型场景

### 场景 A：销售表

- 表级：
  - 销售插件用户可读
  - 销售管理员可写
- 列级：
  - 毛利列仅销售经理可见
  - 内部评语列仅管理员可写
- 行级：
  - 华东销售成员组只看华东客户记录

### 场景 B：审批表

- 表级：审批插件用户可读写
- 视图级：
  - 审批人视图：审批组可访问
  - 发起人视图：普通用户可访问
- 行级：
  - 发起人只看自己单据
  - 审批管理员可看部门内单据

### 场景 C：敏感字段例外

- 列级：`薪资建议` 列默认只读
- 单元格级例外：
  - 当记录属于高管名单时，只允许 HRBP + HR admin 查看

这类例外应通过规则生成，不应人工逐格维护。

## 推荐实现顺序

1. 巩固现有表级 / 视图级 / 列级 / 行级模型
2. 支持成员组作为权限主体
3. 增加“基于成员组/负责人字段”的行级规则
4. 增加列级的批量治理与模板能力
5. 最后再做单元格级例外规则

## 不建议的做法

- 不建议直接开放任意单元格 ACL
- 不建议直接把钉钉部门作为多维表长期权限主体
- 不建议把插件管理员权限和多维表全局管理员权限混在一起
- 不建议绕开成员组单独为每张表再造一套组织树

## 一句话结论

这套平台后续完全可以支持：

- 某些用户/权限组看不到某列
- 某些用户/权限组只能改某些列
- 某些用户/权限组只能看/改某些行
- 少量高敏感单元格例外

但最佳路线是：

- 先做强列级和行级；
- 让成员组成为 ACL 主体；
- 单元格级只做例外层，不做全量自由配置。
