# Attendance Group Fixed-Schedule Managed Provenance Design

Date: 2026-05-28
Branch: `codex/attendance-group-fsd-design-20260528`
Status: design lock only

## 1. Purpose

FS-A through FS-C made the attendance group fixed-schedule path real: preview, apply, conflict safety, and idempotent reruns are landed. The remaining gap is provenance.

Today, a row in `attendance_shift_assignments` does not record whether it came from a manual assignment, a group fixed-schedule apply, an import, or any future producer. Therefore the system cannot safely rebuild or clear "generated by this group schedule" rows without risking manual rows.

This document locks FS-D: managed provenance for group fixed-schedule assignments. It does not implement runtime, routes, migrations, UI, or production writes.

## 4. Design Decision

FS-D should add explicit producer metadata to `attendance_shift_assignments`, not create a second schedule fact table.

Recommended V1 metadata columns:

| Column | Type | Meaning |
| --- | --- | --- |
| `producer_type` | `text null` | Null means unmanaged/manual/legacy. Group fixed-schedule rows use `attendance_group_fixed_schedule`. |
| `producer_ref_id` | `uuid null` | For group fixed schedule, the `attendance_groups.id` that produced the row. |
| `producer_key` | `text null` | Stable business key for the producer/window. |
| `producer_run_id` | `uuid null` | One apply/rebuild run ID for diagnostics and rollback evidence; not the identity key. |

Recommended indexes:

- `(org_id, producer_type, producer_key)`;
- `(org_id, producer_type, producer_ref_id)`.

Recommended constraint:

- either all producer metadata columns are null, or all four are non-null.
- Existing rows remain all-null. New group-generated rows must set the full four-field metadata set.

This is a migration-bearing design. The migration is the reason FS-D must remain a separate explicit opt-in.

## 5. Producer Identity

For group fixed-schedule rows:

```text
producer_type = attendance_group_fixed_schedule
producer_ref_id = groupId
producer_key = attendance_group_fixed_schedule:{groupId}:{shiftId}:{startDate}:{endDate-or-null}
producer_run_id = generated UUID per apply/rebuild run
```

The producer key intentionally includes the selected shift and date window. A different shift or date window is a different managed set.

The producer key must not include member labels, member names, group name, group code, timezone, or UI copy. `userId` remains the only member identity.

## 6. Runtime Semantics To Lock Later

FS-D runtime should be split, not bundled with a larger scheduling editor:

| Slice | Goal | Notes |
| --- | --- | --- |
| FS-D1 | Migration + mapper exposure | Add producer columns and indexes. Existing rows remain unmanaged (`null`). Effective-calendar behavior remains unchanged. |
| FS-D2 | Tag new group fixed-schedule creates | FS-C apply starts populating producer metadata for newly inserted `wouldCreate[]` rows. Existing exact-match rows are never claimed or mutated. |
| FS-D3 | Managed rebuild/clear | Explicit operation can deactivate rows that match the same `producer_type` + `producer_key`; never hard-delete and never touch unmanaged rows. |

Do not combine FS-D with weekly matrix, group punch settings, owner/sub-owner, export/copy, or comprehensive-hours writes.

## 7. Skip, Conflict, And Ownership

FS-C skip/conflict semantics remain intact, but FS-D adds ownership classification:

| Existing row | Classification | Ownership action |
| --- | --- | --- |
| Active exact assignment with same `producer_type` and `producer_key` | `skippedManaged` / already satisfied | No-op; row is managed by this producer. |
| Active exact assignment with null producer metadata | `skippedUnmanaged` / already satisfied | No-op; do not claim, retag, update, or delete later. |
| Active exact assignment managed by a different producer key | `skippedExternalManaged` / already satisfied | No-op; do not claim or delete. |
| Active overlapping non-exact shift assignment | blocking conflict | Apply/rebuild writes zero rows. |
| Active overlapping rotation assignment | blocking conflict | Apply/rebuild writes zero rows. |
| Inactive row | ignored for conflict | Runtime may report it for diagnostics, but must not reactivate silently. |

The key rule: metadata must never be used to bypass the existing overlap safety. It only narrows which generated rows can be rebuilt or cleared later.

## 8. Rebuild And Clear Policy

Managed rebuild/clear must be explicit and transactional.

Recommended rebuild behavior for one producer key:

1. Resolve current group, shift, date window, and full member target set.
2. Lock every target user plus every currently active managed row user for that producer key.
3. Run the same overlap classifier used by preview/apply.
4. If any blocking conflict exists, write zero rows.
5. Create missing target rows with producer metadata.
6. Soft-deactivate active managed rows for the same producer key whose users are no longer in the current target set.
7. Return created/skippedManaged/skippedUnmanaged/skippedExternalManaged/deactivated/blockingConflicts counts.

Recommended clear behavior for one producer key:

- soft-deactivate rows matching `org_id`, `producer_type`, and `producer_key`;
- do not hard-delete;
- do not touch unmanaged rows;
- do not touch rows from another producer key;
- return the deactivated count and a bounded sample.

Soft-deactivation means `is_active=false`. This keeps existing audit shape and ensures effective-calendar no longer consumes those rows.

Changing the shift or date window creates a different `producer_key`. If apply/rebuild for a new key is blocked by active managed rows from the same group fixed-schedule producer but a different key, the runtime should surface that as "clear the existing managed schedule first" rather than as an opaque external conflict.

`producer_run_id` is diagnostic/run evidence. Clear and rebuild ownership is keyed by `producer_key`, not by `producer_run_id`.

## 9. Backfill Policy

No automatic backfill.

Rows created before FS-D remain unmanaged. Even if an old row exactly matches a group fixed-schedule producer key, the runtime must not infer that it is managed. The operator can rerun apply/rebuild; the old exact row will be skipped as unmanaged, not claimed.

This conservative rule prevents accidental ownership of manual rows.

## 10. UI Contract

FS-D runtime may add small status copy to the existing fixed-schedule panel, but must not introduce a weekly matrix or schedule editor.

Allowed UI information:

- whether rows created by this apply are managed by group fixed schedule;
- counts for managed skips, unmanaged skips, external managed skips, created rows, deactivated rows, and blocking conflicts;
- a specific hint when a same-group but different-key managed schedule is blocking a new window/shift;
- a clear/rebuild action only after the matching backend route exists and has tests.

Disallowed UI in FS-D:

- fake disabled weekly matrix controls;
- group-specific punch settings;
- owner/sub-owner controls;
- export/copy controls;
- comprehensive-hours write controls;
- any claim that unmanaged exact rows will be cleared later.

## 12. Explicitly Deferred

These are not unlocked by this design:

- weekly schedule matrix;
- multiple shifts per weekday;
- group-owned rotation generation;
- automatic scheduling of future members without explicit rebuild;
- group-specific punch method;
- owner/sub-owner permissions;
- export/copy group settings;
- comprehensive-hours writes from the group detail;
- use of `attendance_schedule_groups` for attendance-group fixed schedules;
- hard-delete of generated rows.

## 13. Acceptance For This Design

This design is complete when:

- it records why FS-C cannot safely rebuild/delete without provenance;
- it chooses assignment-row producer metadata instead of a second schedule fact table;
- it defines producer type/ref/key/run semantics;
- it states that existing/manual rows are never claimed by inference;
- it defines soft-deactivate, not hard-delete, for managed clear/rebuild;
- it preserves FS-C conflict safety and idempotency semantics;
- it keeps runtime and migration work behind a later explicit opt-in;
- it adds no runtime code, migration, tests, routes, permissions, OpenAPI, UI, ops script, or production writes.
