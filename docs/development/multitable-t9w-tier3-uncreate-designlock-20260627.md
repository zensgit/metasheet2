# T9-W Tier 3 (U-3) un-create — DESIGN-LOCK (RATIFIED · runtime shipped #3335)

**Date:** 2026-06-27 · **Type:** T9-W unsafe-restore write-slice design-lock, narrowed v1. Locks scope + the per-decision bars **before any runtime code**. Docs-only. Unblocks the umbrella lock's `🔒 U-3 (HOLD) … until a separate destructive-delete sign-off` (`multitable-t9-w-unsafe-restore-design-lock-20260626.md`).

## The gap
Config-restore is **revision-scoped** (`config-restore-preview`/`-execute` revert one `revisionId`). `classifyRevert` routes **every `create`-action revision** to `kind:'gated'` (`config-restore.ts:33` — "create/delete = undelete, gated"), and execute returns **422 `RESTORE_NOT_SUPPORTED`** (`univer-meta.ts:7828`). So reverting a config `create` is classified but inert today — the exact T8-1 shape (an inert classifier we turn on behind a flag).

Tier 3 turns "revert a `create` revision" into a real **un-create**: drop the entity that revision created. It is the inverse of Tier 4 (undelete = revert a `delete`) — same `meta_config_revisions` recorder, symmetric — so this lock also sets up Tier 4. **Premise verified:** forward creates DO record `action:'create'` rows (field-create `univer-meta.ts:9672`/`fieldCreateDiff`; view-create `10445`/`11170` via `configCreateDiff`), with `before=null`, `after={full config}`, `changed_keys`=the full config key set. So "revert a create" has a real row to operate on.

## v1 surface — LOCKED NARROW (do not blanket-open un-create)
- **IN:** revert a **single `create` revision** for **`entity_type ∈ {field, view}`** only — the two entity types a real forward create-route records. `field:create` un-create (**lossy** — drops the column's data) + `view:create` un-create (**non-lossy** — view config only, no record data).
- **OUT — each a separate, explicit later decision (NOT opened here):** `delete`→undelete (**Tier 4**, its own line) · **permission** create/delete (permission-revert — held entirely until a per-grant re-grant policy) · `sheet_config` create (1-row-per-sheet; un-creating it is nonsensical) · **bulk / PIT "config reset"** (reconstruct config-at-T and drop everything created after T — a future config-reset slice, the T8-2 analog; **explicitly not this slice**) · cross-base config restore · any FE.

## Locked decisions
- **U3-L1 — Per-revision un-create, NOT PIT-bulk.** Operate on one `revisionId` whose `action='create'`; drop exactly that entity. **No config-at-T reconstruction** (that is the deferred config-reset). This is the load-bearing simplification: the existing route already loads the specific revision's `before`/`after`.
- **U3-L2 — Reuse the REAL forward delete sink (no orphans / no residue).** **field** un-create MUST execute the identical cascade as `DELETE /fields/:fieldId` (`univer-meta.ts:10142–10274`): delete `meta_fields` row + drop `meta_field_auto_number_sequences` + delete `meta_links WHERE field_id` + **strip the column key from every `meta_records.data`** + reorder trailing fields (`order-=1`) with per-shift cascade revisions + clean every view's `filter/sort/group/hidden` config referencing the field with per-view cascade revisions — **all under ONE `batchId`**. **view** un-create MUST execute the identical cascade as `DELETE /views/:viewId` (`11002–11041`). **Mechanism:** extract the inline forward-delete cascade into a shared, **txn-parameterized** helper (`dropFieldCascade(query, …)` / `dropViewCascade(query, …)`) called by BOTH the forward route AND the un-create execute — single source of truth. The extraction MUST be **behavior-preserving for the forward route** (pin with the existing forward-delete tests). Equivalent in-place replication is acceptable ONLY if cascade parity is test-pinned for both callers; extract-and-reuse is preferred.
- **U3-L3 — Records a `delete` revision (forward-only symmetry + audit).** The un-create records the entity's removal as a `delete` revision with `source='restore'` + `restored_from_id` = the reverted `create` revision id (`meta_config_revisions.source` already allows `'restore'`; `restored_from_id` exists). The cascade revisions (order shifts, view cleanups) share the un-create's `batchId`, exactly as the forward delete does.
- **U3-L4 — Opaque HMAC `planHash` preview-identity (NO raw plan field anywhere client-readable).** The config preview-identity is an **HS256 JWT whose payload is client-base64-decodable**, so **no raw plan field** may become a token claim or a preview-response field — that would re-leak precisely what U3-L5 forbids (whether a hidden column has data, the hidden view-reference set, plan details). The un-create drift control is a **single opaque `planHash`** bound as one claim (the existing `baselineHash` claim shape). Its raw inputs are computed **server-side only** at preview AND re-computed server-side at execute, and are the **blast-radius / safety set** — { entity-alive(same id), the **cascade set** (view ids whose config currently references the field), the **order-shift set** (trailing field ids to be reordered), and (field) a **column-data-presence** marker }. These raw inputs are **never serialized** into the token or the response; they only feed the hash.
  - **HMAC-keyed, NOT plain sha256.** The presence marker is ~1 bit of entropy; a plain `sha256` (the `baselineHash` style) over a known structure lets a client who decodes the JWT hash both possibilities and **confirm** whether the hidden column has data — defeating U3-L5. So `planHash = HMAC(server-secret, canonical(raw plan))`: the server key makes it a PRF the client cannot invert or test inputs against.
  - **Excludes cosmetic config (no false-409).** The entity's display config (e.g. `name`) is **not** a plan input, so a benign post-create rename does not change `planHash` → un-create still proceeds. The un-create path therefore does **NOT** apply the existing `driftConflict`/`baselineHash` 409 (those false-trip on rename — they are for update-reverts); `planHash` is its **sole** drift control.
  - **Execute = recompute + compare → ONE generic `PLAN_DRIFT`.** Execute recomputes the raw plan from current DB state, re-derives `planHash`, and on mismatch returns a single generic **409 `PLAN_DRIFT`**. It MUST NOT differentiate "new view reference / new column data / cascade-set changed / entity gone" — a sub-reason is itself an oracle.
- **U3-L5 — Data-loss + no-oracle = UNDISCLOSED, no counts (v1).** Dropping a created **field** hard-drops its column data across all records. Preview MUST NOT surface a per-cell / per-record affected **count** — a count leaks row existence under row-level read-deny, and `canManageFields` does NOT imply unrestricted row read. **v1 strategy = name the entity + state "its column data (all created after T) will be permanently dropped" with NO count.** (The full-read-gate alternative also clears the bar but buys little — choose undisclosed.) **View** un-create is non-lossy → no count concern. This is the **first load-bearing use of U-L8 no-oracle in T9-W**: Tier 1 = config flags, Tier 2 = schema-only/lossless (per #3323's corrected premise), so Tier 3 is where no-oracle actually bites.
- **U3-L6 — Flag + permission floor + typed confirm + atomicity.** New default-off flag **`MULTITABLE_ENABLE_CONFIG_UNCREATE`** (preview AND execute gated → 403 when off). Permission: **retain and require** the route's existing entity-type cap (`field→canManageFields`, `view→canManageViews`) — un-create is delete-equivalent on that entity, **never lowered**. Typed **`confirm:'uncreate'`** (T8-2 destructive pattern; missing → 400). The whole un-create (entity drop + full cascade + delete-revision) runs in **ONE `pool.transaction`**; any guard trip / plan drift → **zero writes**.
- **U3-L7 — Recoverability stated CONSERVATIVELY (no Tier-4 safety-net).** Un-create **hard-drops** the entity and (for field) its column data with **NO v1 undo**. The config-history records a `delete` revision so a *future* Tier 4 slice *could* recreate the field/view **definition** — but the column **values are gone**. Do NOT describe un-create as "recoverable", and do NOT lean on the unbuilt Tier 4 path. (By construction the dropped values are all **post-T** — a field created after T has no pre-T cell values — so the loss is consistent with "restore to T"; it is still irreversible real data entered between T and now.)
- **U3-L8 — Realtime/event parity with forward delete.** Forward field/view delete only invalidates caches (`invalidateFieldCache`/`invalidateViewConfigCache`, `univer-meta.ts:10262`), no realtime config broadcast. v1 un-create **matches that** (same invalidations); a config-change realtime event is explicitly out of v1 (parity, no regression).

## Test matrix (gated on ratification)
flag-off → 403 · permission gate (e.g. a `view`-only actor un-creating a field → 403) · **happy field un-create** (entity gone + column stripped from every record's `data` + `meta_links` gone + auto-number seq gone + trailing fields reordered + every referencing view's config cleaned + `delete` revision recorded, all ONE `batchId`) · **happy view un-create** (view gone + `delete` revision) · **sink-reuse parity** (the extracted `dropFieldCascade`/`dropViewCascade` produces the identical cascade as the forward delete route — asserted for BOTH callers) · **plan-drift → ONE generic `PLAN_DRIFT`** (each of: a new view references the field post-preview / new column data written post-preview / the cascade-set or field-order set changed / entity already gone → ALL return the SAME 409 `PLAN_DRIFT`, no sub-reason) · **NO benign-rename false-409** (post-create rename, then un-create still proceeds — `planHash` excludes cosmetic config; the existing `baselineHash`/`driftConflict` 409 is not applied on the un-create path) · **token/response opacity** (decode the preview JWT + inspect the preview response → assert NO raw cascade set / order-shift set / column-data presence / ids / counts; only the opaque `planHash`) · **no-oracle** (preview exposes no per-cell/row count; an actor under row-deny gets the same undisclosed message AND the same opaque token shape) · typed-confirm (missing → 400; `'uncreate'` → drops) · **single-txn atomicity** (forced mid-cascade failure → zero writes) · audit `delete` revision (`source='restore'`, `restored_from_id`).

## Decisions
- **Scope** = `field:create` + `view:create` both IN (owner 2026-06-27; do NOT narrow back to view-only).
- **Flag name** = `MULTITABLE_ENABLE_CONFIG_UNCREATE`.
- **ACCEPTED by owner 2026-06-27:** no-oracle = **undisclosed / no counts** (U3-L5); recoverability = **hard-drop / no v1 undo, no Tier-4 safety-net** (U3-L7); the **shared delete-helper extraction is sanctioned** provided the forward `DELETE /fields|/views` behavior is pinned unchanged (U3-L2).
- **AMENDED per the #3325 review (this revision):** **U3-L4** rewritten — the un-create drift control is an **opaque HMAC `planHash`**; raw cascade/order-shift/column-data-presence are **server-side-only** digest inputs, never token claims or response fields; execute returns one generic **`PLAN_DRIFT`**. (Resolves the U3-L4↔U3-L5 oracle contradiction.)
