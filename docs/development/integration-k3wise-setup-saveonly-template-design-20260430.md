# K3 WISE Setup Save-Only Pipeline Template Design - 2026-04-30

## Context

The M2 K3 WISE Live PoC safety model requires Save-only execution until the customer explicitly approves Submit/Audit behavior. The backend runner already forwards `pipeline.options.target` into the K3 WISE WebAPI adapter, and the preflight packet already emits:

```json
{
  "options": {
    "target": {
      "autoSubmit": false,
      "autoAudit": false
    }
  }
}
```

The remaining gap was the frontend K3 WISE setup page. Its generated material/BOM pipeline templates did not include the same target policy, so a UI-created pipeline could fall back to the external-system `autoSubmit` / `autoAudit` defaults.

## Change

`apps/web/src/services/integration/k3WiseSetup.ts` now emits explicit Save-only target options in both generated templates:

- material pipeline: `options.target.autoSubmit=false`, `options.target.autoAudit=false`
- BOM pipeline: `options.target.autoSubmit=false`, `options.target.autoAudit=false`

This keeps the setup page aligned with the preflight-generated packet and the runbook's M2 safety boundary.

## Behavior

- The K3 external system can still store `config.autoSubmit` / `config.autoAudit` for future approved flows.
- Pipelines generated by the setup page remain Save-only by default.
- The backend runner continues to pass `pipeline.options.target` to the adapter.
- The K3 adapter continues to own boolean coercion and request-vs-config precedence.

## Non-Goals

- This does not add a UI toggle for Submit/Audit in pipeline templates.
- This does not enable production K3 writes.
- This does not change the preflight script, adapter, or runner contracts.
- This does not claim customer Live PoC PASS; customer GATE and real K3 evidence are still required.
