# DingTalk No-Email Target Guard Development - 2026-05-12

## Summary

This slice hardens the DingTalk P4 no-email manual evidence path so a strict PASS cannot be recorded for the wrong DingTalk directory account.

The change directly addresses the live 142 acceptance finding where the active smoke target was already linked to an admin local user, while a separate `ddzz` no-email account was also present. Without a target match guard, an operator could accidentally record `no-email-user-create-bind` PASS evidence for a different no-email account than the one generated by the smoke session.

## Changed Files

- `scripts/ops/dingtalk-p4-evidence-record.mjs`
- `scripts/ops/compile-dingtalk-p4-smoke-evidence.mjs`
- `scripts/ops/dingtalk-p4-remote-smoke.mjs`
- `scripts/ops/dingtalk-p4-evidence-record.test.mjs`
- `scripts/ops/compile-dingtalk-p4-smoke-evidence.test.mjs`

## Behavior Change

Before:

- `no-email-user-create-bind` PASS required structured admin evidence.
- The evidence contract required `boundDingTalkExternalId`, but it did not enforce equality with the generated smoke target when `targetDingTalkExternalId` was present.
- A manually entered bound id from another DingTalk account could pass if the other admin fields and artifacts were present.

After:

- The evidence recorder rejects `no-email-user-create-bind` PASS when the smoke template has `evidence.adminEvidence.targetDingTalkExternalId` and the submitted `--admin-bound-dingtalk-external-id` is different.
- The strict compiler independently rejects hand-edited JSON with `bound_dingtalk_external_id_mismatch`.
- The generated editable template now includes `targetDingTalkExternalId` in the no-email admin evidence shape.
- Manual checklist text now states that the bound DingTalk external id must match the smoke target.

## Design Notes

The guard is intentionally conditional:

- If `targetDingTalkExternalId` exists, `boundDingTalkExternalId` must match it exactly after trimming whitespace.
- If older evidence templates do not include `targetDingTalkExternalId`, the existing structured evidence contract remains compatible.

This preserves old evidence tooling compatibility while protecting the active remote-smoke workflow that already emits the target id.

## Impact On Current 142 Closeout

This change does not close the current DingTalk live acceptance by itself.

It prevents an unsafe closeout path:

- The active 2026-05-12 smoke target ending in `1174` is already linked to an admin local user.
- The `ddzz` account ending in `9104` is blank-email and linked to a no-email local user, but it is not the active smoke target.
- With this guard, recording `ddzz` as PASS evidence for the active `1174` smoke target is rejected instead of silently accepted.

The remaining valid closeout path is still to use a truly unbound no-email DingTalk directory account as the smoke target, create/bind it through the admin flow, capture redacted artifacts, and finalize the session.

## Security Notes

- No webhook URLs, `SEC` values, JWTs, app secrets, temporary passwords, or full public form tokens were added.
- The code only compares evidence ids already present in the redacted smoke evidence JSON.
- The tests use synthetic ids only.
