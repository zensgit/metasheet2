# DingTalk 组织架构方案收口开发及验证

## 背景

当前 DingTalk 集成已经具备：

- 钉钉目录部门与成员同步到本地目录镜像
- 目录成员绑定 / 自动准入 / 手工准入
- 选定部门投影到平台成员组
- 基于成员组继续做权限、表单与消息治理

但产品层面仍容易让管理员误解为：

- 系统会自动生成一套独立、可编辑的本地组织树
- 或系统支持“本地组织架构”和“钉钉组织架构”双主数据并行维护

这会直接影响后续验收预期，因此需要把当前策略显式写进界面。

## 本轮目标

- 在目录同步管理页明确表达当前组织架构方案边界：
  - 已支持：钉钉目录镜像
  - 已支持：选定部门投影为平台成员组
  - 暂不提供：第二套可编辑本地组织树

## 本轮实现

### 1. 目录页新增“组织架构同步说明”

- 文件：`apps/web/src/views/DirectoryManagementView.vue`
- 在目录集成配置区、成员组同步配置附近新增说明块。
- 文案明确：
  - 当前方案会同步钉钉部门和成员到目录镜像
  - 可将选定部门投影为平台成员组
  - 这些投影用于权限、表单和消息治理
  - 不会自动生成第二套可编辑的本地组织树

### 2. 增加前端回归测试

- 文件：`apps/web/tests/directoryManagementView.spec.ts`
- 在基础渲染用例中增加断言，确保：
  - 页面出现“组织架构同步说明”
  - 页面包含“不会自动生成第二套可编辑的本地组织树”提示

## 产品建议

当前最稳妥的正式方案应当是：

1. 钉钉作为组织来源
2. 系统保留目录镜像表
3. 选定部门投影为平台成员组
4. 权限、表单、消息、治理逻辑都围绕本地用户 + 成员组展开

不建议现在新增一套“本地组织树”产品能力，除非后续出现明确需求：

- 脱离钉钉独立维护组织架构
- 多来源组织合并
- 手工维护部门层级并长期保留
- 将“部门”提升为平台一级核心主数据对象
