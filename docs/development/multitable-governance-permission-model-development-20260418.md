# 多维表治理权限模型开发说明

- 日期：2026-04-18
- Worktree：`.worktrees/dingtalk-member-group-role-sync-20260418`
- 分支：`codex/dingtalk-member-group-role-sync-20260418`

## 目标

把“多维表未来是否能支持表/视图/列/行/单元格级权限，以及如何和钉钉成员组/插件管理员联动”的讨论，收成正式设计资产。

## 本轮完成内容

### 1. 新增正式设计文档

新增：

- `docs/development/multitable-governance-permission-model-design-20260418.md`

这份文档明确了：

- 当前仓库已有的表级 / 视图级 / 列级 / 行级权限基础；
- 为什么推荐先做强列级和行级；
- 为什么单元格级应只做例外层；
- 为什么未来应让 `platform_member_groups` 成为多维表 ACL 主体之一；
- 为什么不应直接把 DingTalk 部门长期作为多维表运行时 ACL 主体。

### 2. 将多维表模型与钉钉 / 插件治理模型对齐

本轮没有新增多维表运行时代码，而是将权限模型明确绑定到现有治理方向：

- 钉钉部门 -> 平台成员组
- 平台成员组 -> 插件管理员范围
- 平台成员组 -> 多维表权限主体

这样后续多维表权限就能和已在推进的钉钉治理线共用同一套人群边界。

### 3. 保持当前实现边界清晰

本轮明确把“未来可做”与“当前已实现”分开：

- 当前已实现：
  - sheet
  - view
  - field
  - record
- 当前未实现：
  - 通用 cell-level ACL
  - 成员组作为多维表权限主体
  - 行级规则构建器

## 为什么这轮先做文档而不是直接改代码

原因有两个：

1. 当前多维表权限并不是空白，而是已经有一套正在工作的 sheet/view/field/record 权限推导。
2. 如果不先把“权限层级、主体模型、与钉钉成员组/插件管理员的关系”定清楚，直接做 cell-level 或 group-subject 扩展，后面很容易返工。

因此这轮先做设计收口，避免后续实现方向跑偏。
