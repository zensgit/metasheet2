# Integration Core K3 WISE Live PoC Evidence Design - 2026-04-25

## Objective

Add a local evidence compiler for the M2 K3 WISE live PoC.

The preflight packet generator proves that the planned test is structurally safe before the operator touches customer systems. The evidence compiler is the matching post-run tool: it turns the generated preflight packet plus the operator's live execution evidence into a redacted PASS / PARTIAL / FAIL report.

The compiler does not call PLM, K3 WISE, SQL Server, or MetaSheet APIs. It only validates and summarizes evidence already collected by the operator.

## New Files

- `scripts/ops/integration-k3wise-live-poc-evidence.mjs`
- `scripts/ops/integration-k3wise-live-poc-evidence.test.mjs`

## Input Contract

The CLI consumes:

- `--packet`: JSON generated by `integration-k3wise-live-poc-preflight.mjs`
- `--evidence`: operator-maintained live evidence JSON
- `--out-dir`: output directory for report JSON and Markdown

Required evidence sections:

- `gate`
- `connections.plm`
- `connections.k3Wise`
- `materialDryRun`
- `materialSaveOnly`
- `deadLetterReplay`
- `rollback`
- `customerConfirmation`

If the preflight packet contains a BOM pipeline, `bomPoC` is also required.

If the preflight packet contains a SQL Server channel, `connections.sqlServer` is recorded as an optional phase. It may pass or be skipped depending on the customer-approved channel.

## Safety Rules

The compiler fails fast when:

- the packet is not Save-only;
- the packet does not explicitly block production writes;
- evidence contains unredacted secret-like fields such as `password`, `token`, `session`, `credential`, `apiKey`, or `authorization`;
- evidence says material Save-only used `autoSubmit=true` or `autoAudit=true`.

The compiler returns `FAIL` when:

- a required phase has status `fail`;
- material Save-only writes fewer than 1 or more than 3 rows;
- material Save-only lacks a `runId`;
- material Save-only has no K3 test record evidence;
- BOM PoC is required and uses the legacy `pipeline.options.source.productId` path.

The compiler returns `PARTIAL` when:

- a required phase is missing, `todo`, `blocked`, or skipped;
- non-fatal warnings are present.

The compiler returns `PASS` only when all required phases pass and no safety issue exists.

## Output Contract

The CLI writes:

- `integration-k3wise-live-poc-evidence-report.json`
- `integration-k3wise-live-poc-evidence-report.md`

The report contains:

- final decision;
- packet safety summary;
- BOM and SQL scope;
- phase table;
- issue list;
- redacted evidence payload.

## Operator Flow

Generate a sample evidence file:

```bash
node scripts/ops/integration-k3wise-live-poc-evidence.mjs --print-sample-evidence
```

Compile the report after live execution:

```bash
node scripts/ops/integration-k3wise-live-poc-evidence.mjs \
  --packet artifacts/integration-live-poc/customer-test/integration-k3wise-live-poc-packet.json \
  --evidence /tmp/k3wise-live-evidence.json \
  --out-dir artifacts/integration-live-poc/customer-test
```

Use the generated Markdown report as the customer-facing PoC evidence summary after checking that no customer secrets are included.

## Non-Goals

- No real customer connectivity.
- No automatic evidence collection from MetaSheet APIs.
- No production rollout approval.
- No M3 UI decision by itself; `PASS` only means the backend live PoC evidence is complete enough to start M3 planning.
