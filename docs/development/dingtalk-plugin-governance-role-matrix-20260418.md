# 钉钉 / 插件治理角色矩阵

- 日期：2026-04-18
- Worktree：`.worktrees/dingtalk-member-group-role-sync-20260418`
- 分支：`codex/dingtalk-member-group-role-sync-20260418`

## 目标

用一页矩阵把下面几件事讲清楚：

1. 普通用户、组织管理员、插件管理员、平台管理员分别能做什么。
2. 同一个用户能否在插件 A 是管理员、在插件 B 是普通用户。
3. 钉钉部门、平台成员组、插件管理员范围三者如何联动。

## 总原则

1. 顶级组织不是普通用户自助创建对象。
2. 普通用户可以被开通一个或多个插件。
3. 插件管理员是**按 namespace 生效**的，不是全局管理员。
4. 插件管理员还必须带范围：
   - 钉钉部门范围；
   - 或平台成员组范围。
5. 钉钉部门继续投影到 `platform_member_groups`，不额外维护第二套本地组织树。

## 角色矩阵

| 角色 | 是否可创建顶级组织 | 是否可创建用户 | 是否可配置钉钉同步 | 是否可管理成员组 | 是否可分配插件角色 | 是否可管理所有插件 |
|---|---|---|---|---|---|---|
| 普通用户 | 否 | 否 | 否 | 否 | 否 | 否 |
| 组织管理员 | 否 | 是 | 是 | 是 | 可在授权边界内操作 | 否 |
| 插件管理员 | 否 | 否 | 否 | 否 | 仅本 namespace、且仅自己范围内 | 否 |
| 平台管理员 | 是 | 是 | 是 | 是 | 是 | 是 |

## 同一用户在不同插件中的身份

这是推荐模型，不是例外。

| 用户 | CRM 插件 | Attendance 插件 | After-sales 插件 | 解释 |
|---|---|---|---|---|
| 用户 A | `crm_admin` | 普通用户 | 未开通 | 在 CRM 是插件管理员，但在 Attendance 只是普通用户 |
| 用户 B | 普通用户 | `attendance_admin` | 普通用户 | 在考勤是管理员，在其他插件只是使用者 |
| 用户 C | 未开通 | 普通用户 | 普通用户 | 只用两个插件，没有任何管理员权限 |

结论：

- “某插件管理员在另一个插件中是普通用户”完全成立；
- 而且应该这样设计；
- 否则插件管理员会变成事实上的全局管理员。

## 插件权限的两层结构

### 第一层：插件开通

决定用户能不能进入某个插件。

例如：

- `crm`：开通
- `attendance`：开通
- `aftersales`：未开通

### 第二层：插件内角色

决定用户进入插件以后能做什么。

例如：

- `crm_user`
- `crm_manager`
- `crm_admin`

所以同一个用户可以是：

- `crm_admin`
- 同时只是 `attendance_user`
- 并且对 `aftersales` 没有任何权限

## 钉钉联动矩阵

| 来源 / 治理对象 | 平台中的承接方式 | 用途 |
|---|---|---|
| 钉钉部门 | 目录镜像 `directory_departments` | 上游组织事实来源 |
| 选定钉钉部门 | 投影为 `platform_member_groups` | 本地治理人群 |
| 平台成员组 | 插件管理员范围 / 默认治理联动 | 约束插件管理员、批量开通插件、补齐默认角色 |
| 用户级 namespace admission | 插件开通层 | 决定用户能不能使用插件 |
| 用户级 namespace 角色 | 插件内角色层 | 决定用户在插件里是什么身份 |

## 推荐的权限边界

### 平台管理员

负责：

- 创建顶级组织 / 工作区；
- 绑定钉钉企业；
- 指定谁是插件管理员；
- 指定插件管理员可管哪些部门 / 成员组；
- 作为最终兜底治理者。

### 组织管理员

负责：

- 创建和维护组织内用户；
- 管理钉钉同步；
- 审核/自动准入同步成员；
- 配置成员组；
- 给用户开通插件；
- 在授权边界内维护插件使用角色。

### 插件管理员

负责：

- 仅在自己的 namespace 下管理角色和开通；
- 仅管理命中自己部门范围或成员组范围的用户；
- 不能创建组织；
- 不能跨插件分配角色；
- 不能越过平台管理员成为全局管理员。

### 普通用户

负责：

- 使用被开通的插件；
- 提交申请；
- 不负责治理其他用户。

## 推荐落地方式

1. 钉钉同步后，先把选定部门投影为平台成员组。
2. 再把成员组作为插件管理员的可治理范围。
3. 插件开通继续走 namespace admission。
4. 插件角色继续走 namespace 下的角色体系。
5. 若要批量治理，则对成员组配置：
   - 默认插件开通；
   - 默认业务角色。

## 一句话结论

- 普通用户可以使用一个或几个插件；
- 同一个用户可以在插件 A 是管理员、在插件 B 是普通用户；
- 插件管理员应当是“按插件 namespace + 按部门/成员组范围”的委派管理员；
- 钉钉部门应继续投影到平台成员组，再由成员组承接插件治理。
