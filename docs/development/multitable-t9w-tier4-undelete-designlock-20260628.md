# T9-W Tier 4 (U-4) config-undelete — DESIGN-LOCK (RATIFIED · runtime shipped #3343)

**Date:** 2026-06-28 · **Type:** T9-W unsafe-restore write-slice design-lock, narrowed v1. Docs-only; locks scope + per-decision bars **before any runtime code**. The inverse of Tier 3 (#3335): Tier 3 reverts a `create` (drop the entity); Tier 4 reverts a **`delete`** (recreate it).

## The gap
`classifyRevert` (`config-restore.ts`) routes every `delete`-action revision to `gated` → execute 422 today (same shape Tier 3 opened for `create`). A `delete` revision carries `before = {full config}`, `after = null`, so reverting it forward-only = **recreate the entity from `before`**. v1 opens the **field/view** subset behind a new default-off flag via a pure `isSupportedUndelete` predicate (mirrors `isSupportedUncreate`).

## The honest constraint (why the umbrella lock said "do NOT fake it")
A field delete (`dropFieldCascade`) hard-drops, in one batch: the `meta_fields` row, **every record's column value** (`data = data - fieldId`), the field's `meta_links`, and its auto-number sequence — and shifts trailing fields `order −= 1` and strips the field from every view's filter/sort/group/hidden. **There is no soft-delete / tombstone.** So an undelete can recreate the field **definition** (name/type/property/order from `before`) but **cannot** recover its values, links, or auto-number counter. v1 is therefore **definition-only**, and says so plainly — it does **not** fake value recovery (the umbrella's bar). View undelete is the clean, non-lossy case (a view holds no record data).

## v1 surface — LOCKED NARROW
- **IN:** revert a single `delete` revision for **`entity_type ∈ {field, view}`**. **view-undelete** = clean recreate of the `meta_views` row from `before`. **field-undelete** = **definition-only** recreate of the `meta_fields` row from `before`, re-inserted at its **original `order`** (trailing fields shift `+1`, the inverse of the delete's `−1`).
- **OUT — each its own later decision:** **permission**-undelete (= permission-revert, the held last slice) · `sheet_config` · **whole delete-batch reversal** (re-adding the field to every view's filter/sort/group + reversing all order-shifts — see U4-L4, clobber-risk; v1 is lazy) · record-data / link / auto-number recovery (**impossible** — definition-only) · cross-base · any FE.

## Locked decisions
- **U4-L1 — Per-revision, NOT whole-batch.** Revert ONE `delete` revision = recreate that one entity. Do **not** auto-reverse the delete's sibling batch (order-shifts of other fields, view-config cleanups). Rationale in U4-L4.
- **U4-L2 — Recreate from `before`, reusing the forward-create shape.** field-undelete = `INSERT meta_fields(id, sheet_id, name, type, property, "order")` from the delete revision's `before`, at `before.order`, **shifting trailing fields `+1`** and recording each shift (the proven forward-create "insert-in-middle" pattern + `recordFieldOrderShifts`, delta `+1`). view-undelete = `INSERT meta_views(...)` from `before` over `VIEW_CONFIG_HISTORY_KEYS`. Reuse/mirror the forward-create INSERT, not a bespoke path — and extract a shared `recreateFieldFromConfig`/`recreateViewFromConfig` if it keeps the forward route behavior-preserving (same discipline as Tier 3's `dropFieldCascade` extraction).
- **U4-L3 — Definition-only, losses stated plainly (no faking).** Column **values**, **`meta_links`**, and the **auto-number counter** are NOT recovered. A re-created **autoNumber** field starts a fresh sequence (documented, not silently "resumed"). **View re-integration is LAZY** — the recreated field is NOT auto-added back into any view's filter/sort/group/hidden (those were cleaned on delete); it reappears in a view only when that view is next edited (same lazy philosophy as Tier 3's inbound-link L4). The preview note says all of this.
- **U4-L4 — Why NOT whole-batch revert (the clobber hazard).** The delete batch carries `before`/`after` on every cascade row, so it is *mechanically* reversible — but blindly reversing it **clobbers any post-delete edits**: a field added after the delete would be order-shifted by a reversal that assumes the old field count; a view edited after the delete would have its filter/sort overwritten by the cascade row's stale `before`. v1 recreates the entity **only** (plus its own order re-insertion against the CURRENT field set), leaving all post-delete state intact. (A future "batch undelete" could opt in with full drift-guarding of every cascade target.)
- **U4-L5 — Plan-hash identity, id-collision + drift, no-oracle.** New **disjoint** `type:'config-undelete-preview'` (an undelete token can't drive an un-create/restore and vice-versa). Bind an opaque **HMAC `recreateHash`** over the SERVER-SIDE plan: `{ idFree (the original id is unoccupied), targetConfig (from `before`), insertOrder, trailingShiftIds }` — raw plan never a claim/response (no-oracle). Execute re-checks under the entity-id lock: original id still free → else **409 `ID_COLLISION`**; plan stable → else **409 `PLAN_DRIFT`** (one generic code). Preview is **undisclosed**: entity name + the definition-only losses note + an id-collision flag if already occupied; **no counts**.
- **U4-L6 — Flag + cap + typed confirm + atomicity.** Default-off **`MULTITABLE_ENABLE_CONFIG_UNDELETE`** (preview + execute → 403 when off). Entity-type cap **required** (`field→canManageFields`, `view→canManageViews`; note `canManageFields===canManageViews===canWrite`, so the floor is canWrite — the same delete-equivalent floor as the forward create/delete). Typed `confirm:'undelete'`. The recreate + order-shift + audit run in **one `pool.transaction`** (zero writes on any guard trip).
- **U4-L7 — Append-only forward `create` revision (`source='restore'`).** The undelete records a `create` revision with `source='restore'`, `restoredFromId = <the delete revision id>` — the exact inverse of Tier 3's `delete`+restore revision, so the undelete is itself auditable and a *future* Tier 3 un-create could drop it again.
- **U4-L8 — Realtime parity.** `invalidateFieldCache`/`invalidateViewConfigCache` post-commit, mirroring forward create/delete; no new config realtime event.

## Test matrix (gated on ratification)
flag-off → 403 · cap (reader) → 403 · **happy view-undelete** (view row recreated from `before` + `create` revision `source='restore'`) · **happy field-undelete** (field row recreated at `before.order`; trailing fields shifted `+1`; create-revision `source='restore'`; **values NOT restored** — assert records still lack the column key; **views NOT auto-re-integrated** — a previously-cleaned view still omits the field) · **definition-only autoNumber** (a recreated autoNumber field has a fresh sequence, not the old counter) · **id-collision** (a new field/view occupies the original id since preview → 409 `ID_COLLISION`, no write) · **plan-drift** (insert-order/trailing set changed since preview → one generic 409 `PLAN_DRIFT`) · **idempotency** (undelete when the entity already exists → reject, never a duplicate) · **token/response opacity** (JWT carries only `recreateHash`; response has no raw plan/counts) · typed-confirm (missing → 400; `'undelete'` → recreated) · **single-txn atomicity** (forced mid-recreate failure → zero writes, no orphan order-shift) · audit `create` revision (`source='restore'`, `restoredFromId`).

## Decisions — for owner ratification
1. **field-undelete = definition-only** (RECOMMENDED: honest + safe) vs whole-batch revert (fuller, clobber-risk per U4-L4). I recommend definition-only.
2. **Order = re-insert at original `order`** (RECOMMENDED: faithful; reuses forward-create insert-in-middle, shifts trailing `+1`) vs append-at-end (simpler, loses position).
3. **View re-integration = LAZY** (RECOMMENDED) vs eager (clobber-risk).
4. **Scope** = view + field both IN (per your sequence). Flag name `MULTITABLE_ENABLE_CONFIG_UNDELETE`.

## Out of scope
permission-undelete (= permission-revert, the held last slice) · `sheet_config` · whole-batch reversal · value/link/auto-number recovery (impossible) · cross-base · FE.
