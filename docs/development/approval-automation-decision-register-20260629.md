### T0-3 — Expose delete_record in the rule editor (safely) · size S

> reviewer note: Anchors all check out and lock_record genuinely supports cross-base, so item-8's lock_record parity is sound (not dead code). Test-plan gaps: (a) test #2 asserts the Save serialize emits exactly {type:'delete_record',config:{}} but does NOT cover the reload/edit round-trip, so the ack-hydration-on-edit blocker above would ship untested; (b) no test asserts the new warning-hint/ack strings are i18n'd; (c) the 'misleading-green' same-base Test Run is well-analyzed but no test asserts the proposed editor hint actually renders. Author already self-declared decisionClean=false; with the missed destructive blast-radius and ack-hydration decisions it remains firmly NOT decision-clean.

- **Q1.** Cross-base delete exposure: should the rule editor expose cross-base delete targeting (targetBaseId/targetSheetId/targetRecordId inputs) in this rung, or only same-base trigger-record delete? The executor already supports cross-base delete and it is savable via raw API today, so this is about UI surface area, not capability.
  - **proposed default:** Same-base trigger-record delete ONLY in v1. Do NOT expose cross-base targeting inputs; cross-base delete stays runtime-capable but editor-unminted, deferred to a separate gated slice. This also keeps Test Run non-destructive (a same-base delete Test Run hits only the synthetic 'test_record' id).
- **Q2.** Anti-misdelete confirm mechanism: what gates a destructive delete rule at authoring time — a required acknowledgement checkbox that blocks Save, a window.confirm on Save, or a typed-phrase confirmation? And must the acknowledgement be kept OUT of the persisted action.config (the serialize map falls through to config: action.config verbatim at MetaAutomationRuleEditor.vue:2980)?
  - **proposed default:** Required acknowledgement checkbox in the delete_record config block that blocks canSave until ticked (mirrors the lock_record checkbox affordance), with copy stating the delete is permanent/non-undoable. The ack is authoring-state held OUTSIDE action.config; the serialize map gets an explicit delete_record branch that emits clean config (same-base → {}), so the ack never persists into the saved rule or any execution.
- **Q3.** Same-base blast radius: a same-base delete_record always deletes the live TRIGGER record (executor uses context.recordId). Should v1 allow targeting a DIFFERENT same-base record (by id / field-path), or stay trigger-record-only?
  - **proposed default:** Trigger-record-only. Same-base delete_record deletes the trigger record (empty config); no per-record id/field-path picker in v1. This caps blast radius to the record that fired the rule.
- **Q4.** Save-time cross-base shape validation: validateCrossBaseWriteConfig (automation-service.ts:283-293) validates update_record ONLY and early-returns for every other type, so a malformed cross-base delete config is currently rejected only at runtime. Extend save-time validation to delete_record (and lock_record) now for fail-closed-at-save parity?
  - **proposed default:** Yes — extend validateCrossBaseWriteConfig to also enforce the cross-base addressing shape for delete_record (and lock_record), independent of the editor exposure decision, because the action is already API-savable. Cheap defense-in-depth that fail-closes a malformed cross-base delete at Save instead of only at run.
- **Q5.** Test Run semantics for delete rules: testRun synthesizes recordId='test_record'/actorId='system' (automation-service.ts:1958-1972), so a same-base delete Test Run hits a non-existent id = 0-row 'success' (misleading green, validates nothing) while a cross-base delete Test Run deletes the real explicit target. Keep Test Run as-is plus an editor hint, block Test Run for destructive actions, or make destructive Test Run a dry-run?
  - **proposed default:** Keep Test Run, add a non-blocking editor/manager hint that Test Run does not delete a real record (synthetic test record). Do NOT block or special-case test for destructive actions in this S rung. (Reinforces the same-base-only default, which is what makes Test Run safe.)
- **Q6.** Authoring permission level: keep parity with other record-mutating actions (canManageAutomation gates create/update/delete rule authoring at univer-meta.ts:15390/15427), or require an elevated capability (e.g. base:admin or a delete-specific grant) to author a destructive delete rule?
  - **proposed default:** Keep canManageAutomation parity. A delete is not privileged above update_record (which can already destroy field data), and authoring is already gated; no new capability in this rung.
- **Q7.** Audit sufficiency: a rule-triggered delete inherits the existing execution-log path (AutomationService records every execution at automation-service.ts:1394/1409) and emits multitable.record.deleted. Is that sufficient, or does a hard-delete-without-undo action need a distinct/elevated audit marker or retention guarantee?
  - **proposed default:** Rely on the existing execution-log path; no new audit table/marker in this rung. Revisit a dedicated destructive-action audit/retention only if the owner wants delete rules tracked separately from other automations.

### T1-1 — Node-level SLA + timeout actions · size M

> reviewer note: Anchors all real and accurate (spot-checked 11: ApprovalActionType 16-24, NodeFieldAccess precedent 32-48, ApprovalNodeConfig 78-90, normalizer rebuild 1037-1049, admin-jump action='jump'+metadata.adminJump 3351/3360, jump-action CHECK migration confirming remind/transfer/jump are already allowed DB actions, checkSlaBreaches 293-306, recordNodeActivation 204-223, sla_scan index 82-86, scheduler tick 135-167, resolveReturnToNode 697-700, resolveAfterApprove 557-580, isReachableDownstream 1708-1713, buildTransferAssignments 770-778). Soundness notes: (1) Decision 2 (storage) is under-scoped on CRITICALITY — the create_approval_metrics migration header (lines 4-9) states metrics writes are 'additive and guarded so metrics errors never fail the parent flow'; placing an authoritative, state-mutation-driving deadline on this best-effort table means a silently-swallowed write disables the timeout entirely. The decision is not just scalar-vs-JSONB-vs-new-table but whether the trigger may live on a fire-and-forget surface at all. (2) Internal inconsistency: Decision 5 lists deadline-reset-on-re-entry as OPEN, but the Runtime/round-trip section states it as DECIDED ('re-entering re-stamps activatedAt, so the deadline resets'). (3) The fired-stamp into node_breakdown JSONB is read-modify-write via mutateBreakdown, racing concurrent recordNodeDecision/recordNodeActivation writes → possible lost stamp → duplicate remind, weakening the single-shot idempotency claim. Test plan is otherwise well-targeted (FAIL-FIRST real-publish wire test correctly guards the normalizer-strip drift); add a test that a jump cascading to terminal approved is itself gated, and a poison-row/unresolvable-target test.

- **Q1.** Which timeout effects are WIRED this rung, and how are the terminal/state-mutating effects gated? auto_approve silently approves a pending request (e.g. spend) and auto_reject silently terminates it — both are spend/security-grade side effects.
  - **proposed default:** Wire remind + transfer + jump this rung. Declare auto_approve/auto_reject in the effect enum but leave them runtime-INERT behind an explicit owner opt-in (env flag APPROVAL_NODE_TIMEOUT_TERMINAL_EFFECTS plus per-template confirmation at publish), mirroring the NodeFieldAccess 'declared-but-do-not-wire' precedent (types/approval-product.ts:32-48). The effect engine refuses to apply a terminal effect until the gate is on.
- **Q2.** Storage model for the per-node deadline the scanner reads: scalar columns on the existing approval_metrics table, JSONB stamped into node_breakdown entries, or a new dedicated approval_node_timeouts table. This is a schema/migration decision.
  - **proposed default:** Default: two nullable scalar columns on the existing approval_metrics table (current_node_deadline_at TIMESTAMPTZ, current_node_timeout_effect TEXT) maintained at node activation/decision, with a partial index mirroring idx_approval_metrics_sla_scan; full per-node history stays in node_breakdown JSONB. Honors the owner lean for columns/JSONB on existing tables over new event tables; the dedicated approval_node_timeouts table is the surfaced indexable alternative.
- **Q3.** Audit-trail discriminator for an auto-fired transfer/jump/auto_approve/auto_reject: reuse the existing approve/reject/transfer/jump record actions with metadata.timeoutEffect=true, or mint new action enum values (CHECK-constraint migration on approval_records).
  - **proposed default:** Reuse existing record actions (approve/reject/transfer/jump) with metadata.timeoutEffect=true, mirroring the adminJump:true metadata precedent (ApprovalProductService.ts:3360). Avoids a CHECK-constraint migration and keeps existing metrics/history filters working.
- **Q4.** Remind cadence: single-shot per node activation, or repeating every N minutes until the node is decided/exited.
  - **proposed default:** Single-shot per node activation, marked done by stamping a fired timestamp; the scan re-check makes a second tick a no-op. Repeating escalation is a separate follow-up rung.
- **Q5.** Deadline clock semantics: plain wall-clock minutes from node activation, or business-hours/working-calendar aware. And on re-entry of a node (return/jump back), does the deadline reset?
  - **proposed default:** Plain wall-clock minutes from the node's activatedAt (already recorded in node_breakdown, ApprovalMetricsService.ts:174-182, 204-223). Deadline RESETS on re-entry because re-entry re-stamps activatedAt. Business-hours clocks are an explicit non-goal this rung.
- **Q6.** Behaviour of timeout effects while the instance is inside a parallel-gateway region (the executor already refuses return/jump inside parallel): allow remind-only, allow all effects, or reject configuration at publish.
  - **proposed default:** Allow remind-only inside a parallel region; reject transfer/jump/auto_* timeout configs on parallel-branch nodes at publish (mirror the existing APPROVAL_JUMP_PARALLEL_BRANCH_TARGET_UNSUPPORTED restriction, ApprovalProductService.ts:3274-3279).
- **Q7.** Identity recorded as the actor for an auto-fired effect (system sentinel id vs the timed-out assignee), and whether auto_reject bypasses a template's rejectCommentRequired policy via a synthetic system comment.
  - **proposed default:** Use a reserved system actor id (e.g. 'system:approval-timeout') recorded in the approval_records actorId + metadata; auto_reject (when later enabled) writes a synthetic system comment so rejectCommentRequired is satisfied without a human comment.
- **Q8.** Timeout granularity unit exposed in the node config (whole minutes vs whole hours). Template SLA is whole hours; node timeouts frequently need finer granularity.
  - **proposed default:** Expose whole MINUTES (afterMinutes, > 0, capped). Finer than template SLA hours; deadline stored as a timestamptz so the scanner SQL is unit-agnostic.

### T1-2 — Inbound webhook endpoint (signed, audited) · size M

> reviewer note: Not decision-clean (author honestly marks 9 open decisions + 4 more above). Test plan is otherwise strong: real mounted route + real DB log, RED-first 404, wire-vs-fixture canonicalization over rawBody, no-side-effect-on-reject, injected-clock replay window, oracle uniformity.

- **Q1.** Signature scheme — does the HMAC bind the timestamp into the signed message, diverging from the existing outbound signPayload which signs body-only? Without binding, a captured (body,signature) replays under any fresh timestamp and replay protection is void.
  - **proposed default:** HMAC-SHA256(secret, "${unixSeconds}.${rawBody}") sent as X-MS-Webhook-Signature: sha256=<hex> + X-MS-Webhook-Timestamp; compared with crypto.timingSafeEqual. Deliberate divergence from outbound signPayload (body-only, webhook-service.ts:630).
- **Q2.** Secret-less webhook.received rules — WebhookReceivedConfig.secret is optional (automation-triggers.ts:49). Are such rules ingestable (accept unsigned) or fail-closed? And should a non-empty secret be enforced at rule-save going forward?
  - **proposed default:** Fail-closed: a rule with no secret is NOT ingestable (returns the uniform 401); additionally require a non-empty secret at rule-save for webhook.received. Never accept unsigned inbound.
- **Q3.** Response-code oracle — secret must be fetched before signature can be checked, so {unknown rule, wrong trigger type, disabled rule, secret-less, stale timestamp, bad signature} all surface at one point. Distinct codes leak rule existence/enabled-state to an unauthenticated caller. Uniform code or differentiated?
  - **proposed default:** One uniform 401 { ok: false } (generic body) for every reject class — no existence/enabled-state oracle.
- **Q4.** Replay-defense depth — signed-timestamp + freshness window only, or add a nonce/dedup store (a new migration)? Freshness-only leaves an in-window replay gap against non-idempotent downstream actions.
  - **proposed default:** Signed-timestamp + a ±300s freshness window for v1; nonce/dedup store DEFERRED. Residual in-window replay risk documented; downstream idempotency is the operator's responsibility.
- **Q5.** Dispatch authority — there is no live user; webhook executions run userless. Confirm they execute under the rule author's stored authority (rule.created_by → context.ruleCreatedBy), the same as scheduled triggers, so the owner signs off that webhook-triggered writes run as the rule author.
  - **proposed default:** Yes — run userless under rule.created_by (→ context.ruleCreatedBy, executor:813), identical to scheduled triggers; surfaced for explicit owner acceptance.
- **Q6.** Secret at rest — trigger_config.secret is stored plaintext in automation_rules.trigger_config JSONB (parity with the outbound webhook secret). Accept plaintext for v1, or require encryption-at-rest (a migration)?
  - **proposed default:** Plaintext parity with the outbound webhook secret for v1, redacted in any rule-read API that surfaces trigger_config; encryption-at-rest deferred.
- **Q7.** Audit-trail persistence (task-named) — are rejected attempts captured only as a structured log line + metric counter, or is a queryable audit record (new table/migration) required to investigate attack patterns? Accepted requests already produce the redacted AutomationExecution row.
  - **proposed default:** Accepted → existing redacted AutomationExecution row; rejected → structured security log line + automation_webhook_rejected_total{reason} metric; no new audit table in v1.
- **Q8.** Sync vs async dispatch — execute inline (like testRun) and return 202 after the executor returns, or enqueue and return 202 immediately? Inline ties up a request worker for slow/abusive actions.
  - **proposed default:** Synchronous inline dispatch (like testRun), respond 202 after the executor returns; no queue in v1.
- **Q9.** Rate limiting + body cap — apply a per-rule inbound rate limit and a per-path body-size cap (below the global 10 MB), and at what limits?
  - **proposed default:** Per-rule rate limit via the existing RateLimitStore (e.g. 60/min/rule) + a 1 MB body cap on the webhook path (below the global 10 MB).
