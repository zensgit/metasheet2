# Metasheet 客户分级实施策略表

Date: 2026-04-03  
Scope: 平台版本在不同客户层级下的实施、升级、灰度、回滚策略

## 1. 文档目标

本文档用于定义不同客户层级在发布实施时应采用的策略，回答以下问题：

1. 哪些客户先升级
2. 哪些客户先启用多维表
3. 哪些客户只适合标准升级
4. 哪些客户需要 blue-green
5. 哪些客户必须先灰度再全量

## 2. 分级原则

客户分级不只看客户规模，还要看以下因素：

1. 业务关键程度
2. 是否接受维护窗口
3. 是否有明确 SLA
4. 是否有现场运维能力
5. 是否同时使用考勤和多维表
6. 是否频繁进行数据导入、报表、附件操作

## 3. 客户层级定义

### L1 内部测试环境

定义：
- 内部研发/测试环境
- 用于快速验证新版本、新配置、新迁移

特点：
- 风险容忍度最高
- 可频繁更新
- 可接受失败和回滚

策略：
- 第一批升级
- 第一批启用新功能
- 默认不需要 blue-green
- 所有高风险功能先在这里暴露

### L2 内部 Pilot 客户

定义：
- 内部试点团队
- 可配合排障和快速反馈

特点：
- 有真实业务，但可控
- 可接受小范围风险
- 可承担早期版本不稳定性

策略：
- 第二批升级
- 多维表优先在此类客户开启
- 建议保留功能开关
- 采用标准升级 SOP 即可
- migration 版本优先在这里验证

### L3 普通客户

定义：
- 正常交付客户
- 无极端高 SLA 要求
- 通常接受低峰短维护窗口

特点：
- 关注稳定
- 现场运维能力一般
- 不适合复杂部署结构

策略：
- 在 pilot 稳定后再升级
- 默认采用标准升级 SOP
- 新功能先灰度，后全量
- 不建议默认上 blue-green
- 升级后必须完成标准 smoke

### L4 关键客户

定义：
- 高价值客户
- 高并发/高活跃现场
- 明确要求高可用或低中断

特点：
- 发布风险高
- 回滚代价高
- 停机窗口小甚至没有

策略：
- 最后升级
- 必须经过 pilot 和普通客户验证
- 建议引入 blue-green
- 建议功能开关与代码部署分离
- 必须有明确 rollback 预案
- 必须在变更窗口内实施

## 4. 部署方式矩阵

| 客户层级 | 推荐部署方式 | 功能启用策略 | 回滚策略 | 备注 |
|---|---|---|---|---|
| L1 内部测试 | 标准升级 | 全量开启 | 快速回滚/重建 | 允许破坏性试验 |
| L2 内部 Pilot | 标准升级 | 灰度开启 | 恢复上一版本 + DB 备份 | 用于真实业务验证 |
| L3 普通客户 | 标准升级 | 分批开启 | 标准回滚 | 低峰执行 |
| L4 关键客户 | Blue-Green | 分级灰度 + 慢放量 | 切流回退优先 | 要求严格验证 |

## 5. 多维表启用策略矩阵

| 客户层级 | 代码部署 | 多维表默认启用 | 建议策略 |
|---|---|---|---|
| L1 | 是 | 是 | 直接启用 |
| L2 | 是 | 是 | 小范围业务先用 |
| L3 | 是 | 否 | 先部署，后按客户启用 |
| L4 | 是 | 否 | 先部署，验证后再启用 |

结论：
- 代码部署不等于功能启用
- L3/L4 不建议一部署就直接全量开放多维表

## 6. 升级顺序建议

每个版本固定按以下顺序推进：

1. L1 内部测试
2. L2 内部 Pilot
3. L3 普通客户
4. L4 关键客户

禁止跳级，例如：
- 未经过 L2 就推 L3
- 未经过 L3 大面积验证就推 L4

## 7. 验收强度矩阵

| 客户层级 | 健康检查 | 人工 smoke | 业务验收 | 观察期 |
|---|---|---|---|---|
| L1 | 必须 | 建议 | 可选 | 短 |
| L2 | 必须 | 必须 | 建议 | 中 |
| L3 | 必须 | 必须 | 必须 | 中 |
| L4 | 必须 | 必须 | 必须 | 长 |

## 8. 回滚要求矩阵

| 客户层级 | 是否必须备份 DB | 是否必须保留上一版本 | 回滚触发门槛 |
|---|---|---|---|
| L1 | 建议 | 建议 | 低 |
| L2 | 必须 | 必须 | 中 |
| L3 | 必须 | 必须 | 中 |
| L4 | 必须 | 必须 | 高但动作必须快 |

## 9. 建议实施分组

### A 组：适合先行升级

特征：
- 内部环境
- pilot 客户
- 业务可控
- 可接受短暂维护

建议：
- 先部署平台版本
- 先启用多维表
- 快速收集反馈

### B 组：适合稳定后跟进

特征：
- 普通客户
- 业务稳定性要求高于试点
- 但仍可接受维护窗口

建议：
- 在 A 组通过后统一批量推进
- 先部署，再按需启用多维表

### C 组：必须谨慎推进

特征：
- 关键客户
- 高 SLA
- 不接受明显中断
- 回滚成本高

建议：
- 最后推进
- 尽量 blue-green
- 必须有明确现场验收和回滚负责人

## 10. 最终建议

推荐你们采用以下分层策略：

1. 代码包始终统一为平台版本
2. 客户按 L1-L4 分层推进
3. 普通客户默认标准升级
4. 关键客户单独建设 blue-green
5. 多维表启用晚于代码部署
6. 每一层都必须有明确验收和回滚标准

## 11. 一句话总结

最好的方式不是“所有客户统一动作”，而是：

- 同一个平台版本
- 不同客户分层推进
- 不同现场采用不同升级方式
- 同一版本下按客户灰度启用功能
