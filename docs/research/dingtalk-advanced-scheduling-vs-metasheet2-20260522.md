## 1. 钉钉「高级排班」12 章能力归纳

按手册章节顺序,每章核心能力一句话:

1. **小组织管理**:按组织架构(部门树)归类考勤组/班次,多门店/工厂场景
2. **排班规则**:13 项子设置(规则名/适用范围/排班周期/合规性上限/一天多班次/入职前禁排/临时划线排班/打卡限制/排班修改/排休修改/请假后排班修改/排班发布工作流/未排班提醒)
3. **排班人**:基于组织架构子集的排班委托,"主负责人→子负责人→排班子管理员→排班人"四级
4. **如何新增排班班组**:班组 = 按工作性质/部门/项目分组的更细排班对象,支持 Excel 模板批量导入/更新
5. **排班周期**:2-31 天周期循环 + 顺延排班(碰到不可排时跳过) + 适用考勤组/班次
6. **调度**:通过 OA 审批的跨考勤组调度。**永久调度**(转考勤组) / **按天调度**(支援几天) / **小时支援**(单日内部分时段);**人员换班**(自己换/换别人/相互替班),审批通过自动重排
7. **高级排班调度设置**:考勤机绑定部门(为按天调度时让员工能在支援部门的考勤机打卡)
8. **排班权限设置**:权限名称 + 成员 + 管理范围(部门/考勤组子集) + 分配权限的细化委托
9. **操作日志**:排班业务操作留痕,审计与合规
10. **综合工时制**:月/季/年总工时上限管控,人员/考勤组维度,**强管控**(超出禁止保存)或**弱管控**(仅提示)
11. **人件费率**(本册篇幅最大,26 图):
    - 员工成本档案(时薪)
    - 预估营业额维护(日历多选,假/休/班标签)
    - 实际营业额维护(手工 / Excel 导入 / 开放接口写入)
    - 排班表实时显示每日人件费(成本)和人件费率(成本/营业额)
    - 报表(日/周/月/季)+ 运营趋势图 + 预估 vs 实际营业差异分析
    - 强管控配置:未维护预估营业额可禁止排班保存
12. **调度考勤报表**:
    - 跨考勤组工时拆分(员工一天在 A+B 考勤组,小时支援场景下分摊)
    - 内置 10 字段(应出勤/出勤/工时/迟到/早退/旷工/加班/考勤结果/历史考勤组) + 支持自定义
    - 应出勤字段拆分(例如把节假日出勤工时单独算)
    - 每日 0 点同步钉钉考勤数据并重算小时支援工时
    - 同步至《钉钉薪酬》(每日 0 点 / 手动)
    - 报表内手动改考勤结果(改后不再被钉钉同步覆盖,可还原)

## 4. 战略观察

### 4.1 这本手册不是"对标 metasheet2 整个考勤模块"的参照系

它是"对标 metasheet2 排班(scheduling)子能力"的参照系。我们在排班一侧能力面**比钉钉高级排班窄**,但 metasheet2 在公式/报表/period rollup 一侧是**钉钉手册根本没讲**的。

### 4.2 真要补,排序

| 优先 | 项目 | 理由 | 阶段一锁定下可否做 |
|---|---|---|---|
| 🟢 高 | **综合工时制**(ch10) | 月/季/年工时上限管控 + 超出禁止保存,**法律合规**刚需(《劳动法》工时上限),工程量小 | ✅ 属 attendance 域内核打磨,**可在 lock 内做**(需独立 RFC 触发) |
| 🟡 中(商业杠杆但工程量大) | **人件费率**(ch11) | 连锁餐饮/零售刚需,需独立产品级模块(员工时薪档案 + 营业额录入 + 趋势图 + API 写入) | ❌ 新战线属性强,**应等 GATE PASS 后评估** |
| 🟡 中 | **排班权限"管理范围"细化**(ch8) + **排班人委托**(ch3) | 多店/多部门场景下,admin 不愿全权,但要重构 RBAC | ⚠️ attendance 域内权限重构,**勉强算内核打磨**;先看客户是否真提 |
| 🟡 中 | **跨考勤组工时拆分**(ch12) | report-records 已铺底,加"按考勤组归属拆分小时"维度即可 | ✅ 内核打磨范围 |
| 🔵 低 | 一天多班次、临时排班、未排班提醒等 ch2 子项 | 各 1-3 天小工程,看具体客户提了再补 | ✅ 内核打磨;按需触发 |
| 🔵 低 | 排班发布工作流 / 排休修改规则 / 请假后排班修改 | 流程类,需要 UX 设计 | ⚠️ 跨 attendance + approval,触发条件:有客户具体提出 |
| ⚫ 不建议 | **调度**(ch6-7)/ 考勤机绑定 / 推钉钉薪酬 | 跟我们 ERP 集成定位错位 | ❌ 不属于 metasheet2 产品线方向 |

### 4.3 一个明确的观察

钉钉「人件费率」(ch11)是这本手册最值得作为**产品启示**的一章 —— 不是因为我们要复制,而是因为它揭示了"**排班 + 成本 + 营收**三联动"是连锁零售/餐饮考勤产品的真实差异化方向。我们在 ERP 集成路线上若延展到零售/餐饮客户,此模块可能是后续 K3 PoC GATE PASS 之后值得**独立立项**的方向。

## 5. 边界约束(K3 PoC stage-1 lock)

memory `project_k3_poc_stage1_lock` 明确:

> 直到客户 GATE PASS,不开新战线 / 不碰 integration-core;**内核打磨允许**。

按此约束:

- 🟢 **可在 lock 内做(内核打磨)**:综合工时制、跨考勤组工时拆分、ch2 子项打磨(一天多班次 / 临时排班 / 未排班提醒)、排班权限细化(若客户提)
- ❌ **应等 GATE PASS 后再评估**:人件费率(新产品模块)、调度链路(跨域跨表)、考勤机绑定(硬件方向)
- ⚪ **不做**:推钉钉薪酬(钉钉生态绑定,与我们 ERP 集成路线相反)

## 6. 复审条件 + 出范围声明

### 何时复审本文档

1. K3 PoC GATE PASS 后,重新评估"人件费率"是否进入下一阶段产品线
2. 任何来自客户的"排班/工时管控"具体需求落地时,优先以此文档对标矩阵作首轮可行性判断
3. 钉钉手册有重大改版(超过 6 个月,2027 Q1 前后)时,重抓 12 章对比 diff

### 不做什么

- ❌ **不立刻基于本文档启动新 slice** —— 此文档是**决策参照**,不是 backlog
- ❌ **不复制钉钉 UI** —— 钉钉手册截图本评估未消化,凭文字盲抄 UI 是高风险
- ❌ **不基于本文档单方面开 RFC** —— 任何后续 RFC 必须由具体客户需求或 GATE PASS 触发,而非对标焦虑触发

### 调研局限性诚实声明

- 钉钉手册的**截图内容(钉钉后台界面、字段名、详细配置项视觉证据)本评估未消化** —— 文字部分共约 9000 字,12 章覆盖率充分,但凡涉及"具体配置面板长什么样、字段叫什么、默认值是什么"的细节,本评估**只能从文字片段推断,可能有偏差**
- 钉钉「高级排班」是付费功能;某些细节可能在不同付费档(假勤高级版 vs 旗舰版)有差异,本评估未区分
